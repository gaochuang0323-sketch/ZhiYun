# 单元测试使用说明

## 1. 环境准备

### 1.1 安装 Ruby + Ceedling

Ceedling 已安装在 `C:\Ruby40-x64\bin\ceedling.bat`，基于 Ruby 4.0。

验证安装：

```bash
"C:\Ruby40-x64\bin\ceedling.bat" version
```

正常输出：
```
Ceedling => 1.0.1-fb1ce6c
CMock => 2.6.0
Unity => 2.6.1
CException => 1.3.4
```

### 1.2 确认 GCC 编译器

测试使用 MinGW GCC 编译（非 arm-none-eabi），验证：

```bash
gcc --version
```

## 2. 运行测试

### 2.1 运行全部测试

```bash
cd /d D:\DESKTOP\Zhiyun_test\ZhiYun
"C:\Ruby40-x64\bin\ceedling.bat" test:all
```

### 2.2 运行单个测试文件

```bash
"C:\Ruby40-x64\bin\ceedling.bat" test:test_voltage_sim
"C:\Ruby40-x64\bin\ceedling.bat" test:test_bms_response_log
"C:\Ruby40-x64\bin\ceedling.bat" test:test_waveform_gen
"C:\Ruby40-x64\bin\ceedling.bat" test:test_dac_safety
"C:\Ruby40-x64\bin\ceedling.bat" test:test_fault_command
"C:\Ruby40-x64\bin\ceedling.bat" test:test_fault_console
"C:\Ruby40-x64\bin\ceedling.bat" test:test_bsp_can
```

### 2.3 每次运行前清理构建目录

Ceedling 不会自动清理旧构建，建议每次运行前删除 build 目录：

```bash
cd /d D:\DESKTOP\Zhiyun_test\ZhiYun
if exist test\build rmdir /s /q test\build
"C:\Ruby40-x64\bin\ceedling.bat" test:all
```

> **注意**：Windows 上有时进程未完全退出，导致 `Permission denied` 错误。此时需要：
> ```bash
> taskkill /f /im ruby.exe
> taskkill /f /im gcc.exe
> taskkill /f /im test_bsp_can.out
> ```

## 3. 理解测试结果

### 3.1 成功输出示例

```
--------------------
OVERALL TEST SUMMARY
--------------------
TESTED:  59
PASSED:  59
FAILED:   0
IGNORED:  0
```

### 3.2 失败输出示例

```
-------------------
FAILED TEST SUMMARY
-------------------
[test/test_waveform_gen.c]
  Test: test_StartSine_valid_params_succeeds
  At line (85): "Expected 0 Was 1"
```

### 3.3 常见失败原因

| 错误信息 | 原因 |
|---------|------|
| `Expected X Was Y` | 断言值不匹配，检查被测函数逻辑 |
| `Called fewer/more times than expected` | CMock 期望调用次数不匹配 |
| `Function called with unexpected argument` | CMock 期望参数与实际不匹配 |
| `undefined reference to 'Xxx'` | 链接错误，缺少桩函数或源码文件 |
| `Permission denied` | 上次运行的进程残留，需 taskkill |

## 4. 测试框架架构

### 4.1 目录结构

```
test/
├── test_xxx.c              # 测试用例源文件（以 test_ 前缀命名）
├── support/
│   ├── stm32h7xx_hal.h     # HAL 库桩（模拟 STM32 HAL API）
│   ├── main.h              # main.h 桩（提供引脚宏定义）
│   ├── dac81416.h          # DAC 驱动桩
│   ├── fdcan.h             # FDCAN 驱动桩（含全部类型定义）
│   ├── cmsis_os2.h         # RTOS 桩
│   ├── spi.h               # SPI 桩
│   ├── usart.h             # USART 桩
│   ├── stubs.h             # 桩函数声明
│   └── stubs.c             # 桩函数实现（HAL_GetTick, DAC_WriteChannel, FDCAN ...）
├── build/                  # 构建产物（自动生成，可删除）
└── project.yml             # 已废弃，实际配置在根目录 project.yml
```

### 4.2 项目配置文件 `project.yml`

关键配置：

```yaml
:paths:
  :source:
    - App/Src/**        # 源码搜索路径
    - BSP/Src/**
  :include:
    - App/Inc/**        # 头文件搜索路径
    - BSP/Inc/**
    - Core/Inc/**
    - test/support/**   # 桩头文件路径（覆盖真实 HAL）

:files:
  :source:
    # 白名单：只编译这些源文件（排除 waveform_engine 等硬件紧耦合文件）
    - App/Src/bms_response_log.c
    - App/Src/dac_safety.c
    - App/Src/voltage_sim.c
    - App/Src/waveform_gen.c
    - App/Src/fault_command.c
    - App/Src/fault_console.c
    - BSP/Src/bsp_can.c
    - BSP/Src/bsp_sram.c

:defines:
  :test:
    - TEST
    - STM32H743xx
    - USE_HAL_DRIVER
    - __weak=                    # 消除 __weak 关键字
    - "__DSB()="                 # 消除 DSB 指令
    - "__ISB()="
    - "__DMB()="
    - "__disable_irq()="
    - "__enable_irq()="
    - "__BKPT()="
```

## 5. 三种测试模式

### 5.1 纯逻辑测试（无硬件依赖）

适用于纯算法模块，无需桩函数：

```c
// test_voltage_sim.c
#include "unity.h"
#include "voltage_sim.h"

void test_MillivoltsToDacCode_returns_0xFFFF_for_max(void) {
    TEST_ASSERT_EQUAL_UINT16(0xFFFF,
        VoltageSim_MillivoltsToDacCode(VOLTAGE_SIM_MAX_MV));
}
```

### 5.2 使用手工桩函数的测试

适用于依赖 HAL、GPIO 等硬件的模块：

```c
// test_bsp_can.c
#include "unity.h"
#include "bsp_can.h"
#include "stubs.h"   // 包含 HAL_SetTick, FDCAN_SetRxData 等

void setUp(void) {
    FDCAN_Reset();
    BspCan_Init();  // 使用桩中的 HAL_FDCAN_xxx 函数
}
```

### 5.3 使用 CMock 自动模拟的测试

适用于需要验证模块间交互的测试：

```c
// test_dac_safety.c
#include "unity.h"
#include "dac_safety.h"
#include "mock_waveform_gen.h"   // CMock 自动生成
#include "mock_voltage_sim.h"

void test_EmergencyStop_calls_dependencies(void) {
    // 设置期望：WaveformGen_Stop 被调用 1 次
    WaveformGen_Stop_Expect();
    // 设置期望：VoltageSim_ClearAllFaults 被调用，返回 HAL_OK
    VoltageSim_ClearAllFaults_ExpectAndReturn(HAL_OK);

    DacSafety_EmergencyStop(3000);
}
```

## 6. 编写新测试

### 6.1 创建测试文件

在 `test/` 目录下创建 `test_模块名.c` 文件：

```c
#include "unity.h"
#include "你要测试的模块.h"
#include "stubs.h"    // 如果需要桩函数

void setUp(void) {
    // 每个测试用例前执行，用于初始化
}

void tearDown(void) {
    // 每个测试用例后执行，用于清理
}

void test_功能描述(void) {
    // 准备
    // 执行
    // 断言
    TEST_ASSERT_EQUAL_UINT16(100, result);
}
```

### 6.2 测试命名规范

- 函数名以 `test_` 开头
- 文件名以 `test_` 开头
- 测试函数名应描述测试场景：`test_功能_条件_期望`

### 6.3 常用断言宏

```c
TEST_ASSERT_EQUAL(expected, actual)
TEST_ASSERT_EQUAL_UINT8(a, b)
TEST_ASSERT_EQUAL_UINT16(a, b)
TEST_ASSERT_EQUAL_UINT32(a, b)
TEST_ASSERT_EQUAL_STRING("expected", actual_str)
TEST_ASSERT_NOT_NULL(ptr)
TEST_ASSERT_TRUE(condition)
TEST_ASSERT_FALSE(condition)
TEST_ASSERT_UINT16_WITHIN(delta, a, b)  // 允许误差范围
TEST_ASSERT_EQUAL_MEMORY(expected, actual, len)
```

### 6.4 CMock 常用 API

```c
// 期望函数被调用（不检查参数）
Func_Expect();

// 期望函数被调用，并指定返回值
Func_ExpectAndReturn(retval);

// 期望函数被调用，指定所有参数和返回值
Func_ExpectAndReturn(arg1, arg2, ..., retval);

// 忽略参数检查
Func_IgnoreArg_argName();

// 忽略所有调用（不检查次数）
Func_Ignore();
Func_IgnoreAndReturn(retval);

// 允许任意参数
Func_ExpectAnyArgsAndReturn(retval);
```

## 7. 添加桩函数

### 7.1 在 stubs.c 中添加

对于 HAL 函数或硬件驱动函数，在 `test/support/stubs.c` 中添加实现：

```c
// 示例：添加 TIM 桩函数
HAL_StatusTypeDef HAL_TIM_Base_Start(TIM_HandleTypeDef *htim) {
    (void)htim;
    return HAL_OK;
}
```

### 7.2 在 stubs.h 中声明

```c
// stubs.h
void TIM_SetStartResult(HAL_StatusTypeDef result);
```

### 7.3 添加桩头文件

当被测模块 `#include` 了系统头文件时，需要在 `test/support/` 下创建同名桩头文件：

- `stm32h7xx_hal.h` — 提供 `HAL_StatusTypeDef`、`GPIO_PinState` 等基础类型
- `fdcan.h` — 提供 `FDCAN_HandleTypeDef`、`FDCAN_TxHeaderTypeDef` 等
- `cmsis_os2.h` — 提供 `osThreadId_t` 等
- `spi.h` — 提供 `SPI_HandleTypeDef` 等

## 8. 覆盖率报告

### 8.1 生成覆盖率报告

```bash
"C:\Ruby40-x64\bin\ceedling.bat" gcov:all
```

报告生成在 `test/build/artifacts/gcov/` 目录下，包含 HTML 文件。

### 8.2 解读覆盖率

- **HtmlBasic** 报告用颜色标注：绿色=已覆盖，红色=未覆盖
- 阈值设置：`html_medium_threshold: 75`，`html_high_threshold: 90`
- 注意：桩文件和测试代码本身不纳入覆盖率统计

## 9. CI 集成

`.github/workflows/ci.yml` 已配置 GitHub Actions：

```yaml
name: CI
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: ruby/setup-ruby@v1
      - run: gem install ceedling
      - run: sudo apt-get install -y gcc
      - run: pip install gcovr
      - run: ceedling test:all
      - run: ceedling gcov:all
      - uses: actions/upload-artifact@v4
        with:
          name: coverage-report
          path: test/build/artifacts/gcov/
```

每次 Push 到 main/master/develop 分支或创建 PR 时自动运行全部测试。
