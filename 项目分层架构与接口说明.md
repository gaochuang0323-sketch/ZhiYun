# ZhiYun BMS 故障模拟器 — 项目架构说明书

本文档面向项目管理者，用通俗语言说明当前固件项目的整体架构、系统使用的设计模式、各模块的职责与数据传递、以及当前开发进度。

---

## 一、项目概况

当前系统是一台 **BMS 故障模拟器**，核心功能是**代替真实的 13 串电池模组**，向 BMS 注入可精确控制的故障信号（过压、欠压、压差），并记录 BMS 的响应时间。

| 项目      | 说明                                |
| ------- | --------------------------------- |
| 主控芯片    | STM32H743（ARM Cortex-M7, 480MHz）  |
| 操作系统    | FreeRTOS 实时操作系统（CMSIS-RTOS V2 接口） |
| 以太网协议栈  | LwIP（轻量级 TCP/IP 协议栈）              |
| 电压输出芯片  | DAC81416（16 位精度，16 通道，SPI 接口）     |
| CAN 通信  | FDCAN1 外设，Classic CAN 模式，500kbps  |
| 串口      | USART1，115200 波特率                 |
| 外部 SRAM | 1MB，16 位总线，通过 FMC 接口连接            |

---

## 二、系统的核心设计模式：多状态机架构

整个系统采用**多状态机架构**——不是用一个庞大的状态机控制一切，而是每个功能模块内部有自己独立的小状态机，各自管理自己的行为。

### 系统中有哪些独立的状态机？

| 模块           | 状态机名称            | 管理什么                 |
| ------------ | ---------------- | -------------------- |
| voltage_sim  | **故障状态机** × 13 路 | 每路电芯的故障注入、保持、恢复过程    |
| dac_safety   | **安全状态机** × 1    | DAC 正常/报警/急停/释放的状态切换 |
| ethernetif   | **PHY 状态机** × 1  | 以太网 PHY 的复位、扫描、链路检测  |
| waveform_gen | **波形状态机** × 1    | 方波/正弦波的激活和停止         |

下面详细说明每个状态机的状态和转换条件。

---

### 2.1 故障状态机（voltage_sim）—— 核心状态机

这是系统中最复杂的状态机，**每路电芯独立运行一个**（共 13 个实例）。每个实例管理该路从"正常"到"故障"再到"恢复"的完整过程。

```
                   用户执行 "ov 7 4500 100 3000"
                          │
                          ▼
 ┌──────────────────────────────────────────────────────────┐
 │     ① 故障上升阶段（RAMPING）                              │
 │     ──────────────────────────────────────                │
 │     行为：每 1ms 按斜率（3000mV/ms）逐步升压               │
 │     从当前电压 → 向目标电压（4500mV）靠近                  │
 │     结束条件：电压到达目标值                                │
 └──────────────────┬───────────────────────────────────────┘
                    │ 电压到达目标值
                    ▼
 ┌──────────────────────────────────────────────────────────┐
 │     ② 故障保持阶段（HOLDING）                              │
 │     ──────────────────────────────────────                 │
 │     行为：保持电压不变（持续输出 4500mV）                   │
 │     结束条件：持续时间到（100ms）                           │
 └──────────────────┬───────────────────────────────────────┘
                    │ 保持时间到
                    ▼
 ┌──────────────────────────────────────────────────────────┐
 │     ③ 故障恢复阶段（RESTORING）                            │
 │     ──────────────────────────────────────                 │
 │     行为：每 1ms 按斜率逐步降压                             │
 │     从目标电压（4500mV）→ 向恢复电压（故障前电压）靠近       │
 │     结束条件：电压回到恢复值                                 │
 └──────────────────┬───────────────────────────────────────┘
                    │ 电压回到恢复值
                    ▼
 ┌──────────────────────────────────────────────────────────┐
 │     ④ 空闲状态（NONE）                                     │
 │     ──────────────────────────────────────                 │
 │     行为：什么都不做，等待下一次故障指令                      │
 └──────────────────────────────────────────────────────────┘
```

**代码中如何实现这 4 个状态？** 用一组标志变量组合判断：

| 标志变量                                    | 作用                                         |
| --------------------------------------- | ------------------------------------------ |
| `cellFaultInfo[cell].type`              | 是否有故障（NONE / OVER_VOLTAGE / UNDER_VOLTAGE） |
| `cellFaultInfo[cell].targetMillivolts`  | 故障目标电压                                     |
| `cellFaultInfo[cell].restoreMillivolts` | 故障恢复电压（故障前的正常值）                            |
| `faultRestoring[cell]`                  | 是否进入恢复阶段（0=上升/保持, 1=恢复）                    |
| `faultHoldStartedMs[cell]`              | 进入保持阶段的时间戳                                 |
| `faultLastUpdateMs[cell]`               | 最近一次更新电压的时间戳                               |

**状态判断逻辑（在 VoltageSim_Process 中每 1ms 执行）：**

```text
进入 Process()
  │
  ├─ type == NONE → 跳过该路（没有故障）
  │
  ├─ 当前电压 != 目标电压（target 或 restore 取决于 faultRestoring 标志）
  │    └─ 按斜率计算下一步电压 → 写入 DAC → 标记输出已改变
  │
  ├─ 电压刚到达 target 且 faultRestoring == 0（刚进入保持阶段）
  │    └─ 记录 faultHoldStartedMs = 当前时间
  │
  ├─ voltage == target 且 保持时间已到（faultRestoring == 0）
  │    └─ 设置 faultRestoring = 1（切换到恢复阶段）
  │
  └─ voltage == restore 且 faultRestoring == 1（恢复完成）
       └─ 清除故障记录，回到 NONE 状态
```

### 2.2 DAC 安全状态机（dac_safety）

这个状态机管理系统的安全保护行为。它监听 DAC 芯片的报警引脚，一旦发现异常就自动进入急停。

```
    上电初始化
        │
        ▼
┌──────────────────────────────────────────────┐
│  ① 正常运行（NORMAL）                         │
│  ─────────────────────                         │
│  行为：每 50ms 读取一次 nALMOUT 引脚            │
│  引脚为高电平 → 正常，继续监测                   │
│  触发条件：检测到引脚变低电平                    │
└──────────────────┬───────────────────────────┘
                   │ nALMOUT = 低电平（DAC 报警）
                   ▼
┌──────────────────────────────────────────────┐
│  ② 急停锁存（EMERGENCY）                      │
│  ─────────────────────                         │
│  行为：                                        │
│  1. 停止所有波形                               │
│  2. 清除所有故障                               │
│  3. 13 路全部设置为安全电压（如 3200mV）        │
│  4. 设置 emergencyActive = 1                  │
│  5. 锁存报警（alarmLatched = 1）              │
│                                               │
│  影响：所有修改 DAC 输出的命令都被拒绝           │
│  触发条件：用户执行 "safe release"              │
└──────────────────┬───────────────────────────┘
                   │ safe release
                   ▼
┌──────────────────────────────────────────────┐
│  ③ 释放（RELEASE）→ 回到正常运行               │
│  ─────────────────────                         │
│  行为：                                        │
│  1. 设置 emergencyActive = 0                  │
│  2. 清除报警锁存                               │
│  效果：恢复正常控制                             │
└──────────────────────────────────────────────┘
```

**注意：** 从 EMERGENCY 状态只能通过手动 `safe release` 回到 NORMAL，不能自动恢复。这是安全设计的核心——DAC 报警一定是硬件出了问题，必须人工检查确认后才能继续使用。

### 2.3 PHY 状态机（ethernetif）

这个状态机管理以太网 PHY 芯片（LAN8720A）的整个生命周期，从上电复位到正常收发，以及热插拔处理。

```
    上电 / 网线插入
        │
        ▼
┌──────────────────────────────────────────────┐
│  ① 扫描寻找 PHY（PHY_SCAN）                   │
│  ─────────────────────                         │
│  行为：每 100ms 轮询 MDIO 总线                 │
│  扫描地址 0~31，读取 PHY ID1 寄存器            │
│  结束条件：找到 ID 匹配的 PHY                  │
│  如果找不到：每 5 秒打印一次"PHY not found"     │
└──────────────────┬───────────────────────────┘
                   │ 找到 PHY，缓存地址
                   ▼
┌──────────────────────────────────────────────┐
│  ② 软件复位（PHY_RESET）                      │
│  ─────────────────────                         │
│  行为：写 BMCR 寄存器 bit15 = 1               │
│  每 10ms 检查复位完成位是否为 0                │
│  超时时间：500ms                               │
│  如果超时：打印错误，等待下次重试               │
│  结束条件：复位完成                            │
└──────────────────┬───────────────────────────┘
                   │ 复位成功
                   ▼
┌──────────────────────────────────────────────┐
│  ③ 链路检测（LINK_DETECT）                    │
│  ─────────────────────                         │
│  行为：每 100ms 读取 BSR 寄存器                │
│  技巧：读两次（latch-low 效应避免方案）         │
│  如果链路 up → 进入 LINK_UP                   │
│  如果链路 down → 继续轮询                      │
└──────────────────┬───────────────────────────┘
                   │ 链路 up
                   ▼
┌──────────────────────────────────────────────┐
│  ④ 链路已连接（LINK_UP）                       │
│  ─────────────────────                         │
│  行为：                                        │
│  1. 读取 PHY 协商速度/双工模式                  │
│  2. 配置 MAC 控制器匹配                        │
│  3. 启动以太网 DMA 收发                        │
│  4. 通知 LwIP 网络就绪                         │
│  5. 打印 "[eth] link up"                      │
│                                               │
│  后续：每 100ms 继续检测链路                    │
│  如果链路 down → 进入 LINK_DOWN               │
└──────────────────┬───────────────────────────┘
                   │ 网线拔出 / 链路断开
                   ▼
┌──────────────────────────────────────────────┐
│  ⑤ 链路断开（LINK_DOWN）                      │
│  ─────────────────────                         │
│  行为：                                        │
│  1. 停止以太网 DMA 收发                        │
│  2. 通知 LwIP 网络断开                         │
│  3. 清除 PHY 地址缓存（支持热插拔）             │
│  4. 打印 "[eth] link down"                    │
│                                               │
│  下一跳：回到 PHY_SCAN 重新扫描                 │
└──────────────────────────────────────────────┘
```

### 2.4 波形状态机（waveform_gen）

这个状态机管理波形输出的启停，状态比较简单：

```
    上电初始化
        │
        ▼
┌──────────────────────────────────────────────┐
│  ① 空闲（IDLE）                               │
│  ─────────────────────                         │
│  行为：每 1ms 调用 Process() 什么都不做         │
│  触发条件：收到 wave square / wave sine 命令   │
└──────────────────┬───────────────────────────┘
                   │ 启动波形
                   ▼
┌──────────────────────────────────────────────┐
│  ② 波形输出中（ACTIVE）                        │
│  ─────────────────────                         │
│  行为：每 1ms 计算波形当前采样点 → 写 DAC      │
│  方波：在 lowMv 和 highMv 之间切换              │
│  正弦波：查表计算 (offset + amplitude × sin)   │
│  触发条件：收到 wave stop 命令                  │
│            或执行 normal/set/ov/uv/clear       │
└──────────────────┬───────────────────────────┘
                   │ 停止
                   ▼
              回到空闲（IDLE）
```

## 三、各状态机的共性特征

| 特征                      | 说明                                                                                                                    |
| ----------------------- | --------------------------------------------------------------------------------------------------------------------- |
| **事件驱动**                | 每个状态机都是由"事件"触发状态切换的。事件可以是：用户命令、硬件中断、定时器超时、外部信号变化                                                                      |
| **每个周期检查一次**            | 所有状态机都在 defaultTask 的 1ms 循环中被检查，而不是用独立线程阻塞等待                                                                         |
| **状态扁平化**               | 每个状态机的状态数量控制在 3~5 个，没有嵌套状态，易于理解和调试                                                                                    |
| **用标志位而不是 switch-case** | 代码中并没有显式的 `switch(state)`。状态通过多个标志变量的组合来判断（例如 `faultRestoring + type + voltage == target` 共同决定当前处于哪个阶段），这种方式更灵活但可读性稍差 |
| **独立实例化**               | 故障状态机有 13 个独立实例（每路电芯一个），互不影响                                                                                          |

## 四、软件分层架构

整个软件从底层硬件到上层应用分为 5 层，每层只和相邻层打交道：

```text
┌──────────────────────────────────────────────────────────────┐
│                    控制入口层                                   │
│  串口（fault_console） / TCP（tcp_json_server） / CAN          │
│  职责：接收外部输入，转成统一格式交给下一层                      │
└──────────────────┬───────────────────────────────────────────┘
                   │
┌──────────────────▼───────────────────────────────────────────┐
│                    命令解析层（fault_command）                   │
│  职责：把收到的文本或 JSON 翻译成"做什么+参数"，调用业务层        │
│  例如：收到 "set 5 3300" → 调用"设置第5路电压为3300mV"          │
│        收到 {"cmd":"ov","cell":7} → 调用"第7路过压"             │
└──────────────────┬───────────────────────────────────────────┘
                   │
┌──────────────────▼───────────────────────────────────────────┐
│                    业务逻辑层（状态机都在这一层）                   │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────────┐  │
│  │ 电压模拟  │  │ 波形生成  │  │ 安全保护  │  │ 响应时间记录  │  │
│  │故障状态机 │  │波形状态机│  │安全状态机 │  │ （非状态机）  │  │
│  │ ×13路    │  │          │  │          │  │              │  │
│  └──────────┘  └──────────┘  └──────────┘  └──────────────┘  │
│  职责：计算电压、生成波形、监测报警、记录 CAN 响应时间           │
└──────────────────┬───────────────────────────────────────────┘
                   │
┌──────────────────▼───────────────────────────────────────────┐
│                    硬件驱动层                                     │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────────┐  │
│  │ DAC81416 │  │ CAN BSP  │  │ ETH BSP  │  │ SRAM BSP     │  │
│  │  驱动     │  │          │  │PHY状态机 │  │              │  │
│  └──────────┘  └──────────┘  └──────────┘  └──────────────┘  │
│  职责：封装硬件操作细节，向上层提供简单接口                      │
└──────────────────┬───────────────────────────────────────────┘
                   │
┌──────────────────▼───────────────────────────────────────────┐
│               HAL 芯片外设层（CubeMX 自动生成）                  │
│  SPI1 / GPIO / USART1 / FDCAN1 / ETH / FMC / FreeRTOS / LwIP │
│  职责：初始化芯片外设，提供 HAL 库函数                          │
└──────────────────────────────────────────────────────────────┘
```

## 五、每个模块的输入、处理、输出

### 5.1 命令执行模块（fault_command）

| 项目       | 说明                                                           |
| -------- | ------------------------------------------------------------ |
| **输入**   | 一行文本（如 `"ov 7 4500 100 3000"`）或一行 JSON（如 `{"cmd":"status"}`） |
| **处理流程** | 判断是文本还是 JSON → 解析参数 → 检查参数合法性 → 调用对应的业务函数                    |
| **输出**   | 调用电压模拟/波形/安全保护等业务层的函数，并将执行结果通过回调函数返回                         |
| **文件**   | `App/Src/fault_command.c`（约 1700 行代码）                        |

### 5.2 串口控制台模块（fault_console）

| 项目       | 说明                                                 |
| -------- | -------------------------------------------------- |
| **输入**   | USART1 串口收到的字节流（用户打字输入）                            |
| **处理流程** | 每 1ms 检查串口是否有新字节 → 拼接成完整一行（遇到回车结束）→ 交给命令解析模块       |
| **输出**   | 调用 `fault_command` 执行命令，命令执行结果的文本通过 `printf` 输出到串口 |
| **保护机制** | 每次轮询结束后让出 CPU（`osDelay(1)`），避免独占 CPU 导致其他任务饿死      |
| **文件**   | `App/Src/fault_console.c`                          |

### 5.3 TCP JSON 服务模块（tcp_json_server）

| 项目       | 说明                                            |
| -------- | ------------------------------------------       |
| **输入**   | TCP 客户端通过端口 5000 发送的 JSON 文本行              |
| **处理流程** | 接受客户端连接 → 逐行读取 JSON → 交给命令解析模块 → 结果写回客户端   |
| **输出**   | JSON 格式的响应（如 `{"ok":true,"cmd":"status"}`） |
| **限制**   | 同一时间只能服务一个客户端（简化设计）                        |
| **文件**   | `App/Src/tcp_json_server.c`                |

### 5.4 电压模拟模块（voltage_sim）—— 包含 13 路故障状态机

| 项目       | 说明                                                                      |
| -------- | ----------------------------------------------------------------------- |
| **输入**   | 来自命令层的指令（设置电压、注入故障、清除故障），以及系统当前时间 tick                                  |
| **处理流程** | 每 1ms 运行 13 路故障状态机 → 判断每路处于什么阶段（空闲/上升/保持/恢复）→ 按斜率计算当前应输出的电压 → 写入 DAC 芯片 |
| **输出**   | 13 路 DAC 通道的电压值（对应 C01~C13 电芯电压）                                        |
| **关键参数** | 电压范围 500~5000mV，每通道独立校准 offset（-500~+500mV），故障支持斜率（mV/ms）和持续时间（ms）      |
| **文件**   | `App/Src/voltage_sim.c`                                                 |

### 5.5 波形生成模块（waveform_gen）—— 包含波形状态机

| 项目       | 说明                                                                     |
| -------- | ---------------------------------------------------------------------- |
| **输入**   | 来自命令层的指令（启动方波/正弦波、停止波形），以及系统当前时间 tick                                  |
| **处理流程** | 每 1ms 检查波形状态机 → 如果激活则计算当前采样点的电压值 → 方波在高低电平之间切换，正弦波用查表法计算 → 把计算结果写入 DAC |
| **输出**   | 1 路（单通道）或 13 路（全通道）的波形电压值                                              |
| **当前限制** | 软件定时 1ms 刷新，适合低频验证（< 100Hz），高频时抖动较大                                    |
| **文件**   | `App/Src/waveform_gen.c`                                               |

### 5.6 DAC 安全保护模块（dac_safety）—— 包含安全状态机

| 项目       | 说明                                                                                   |
| -------- | ------------------------------------------------------------------------------------ |
| **输入**   | DAC 芯片的 nALMOUT 引脚电平，来自命令层的急停/释放指令                                                   |
| **处理流程** | 每 1ms 检查安全状态机 → 实际每 50ms 读取一次硬件引脚 → 如果检测到低电平（报警）→ 自动进入急停状态，停止波形、清除故障、设置安全电压 → 锁存急停状态 |
| **输出**   | 安全状态标志（是否急停、是否报警、报警次数等），控制其他模块是否允许修改 DAC 输出                                          |
| **安全策略** | 急停锁存期间，所有改变电压的命令（normal/set/ov/uv/diff/wave/cal/clear）都会被拒绝，直到手动执行 safe release      |
| **文件**   | `App/Src/dac_safety.c`                                                               |

### 5.7 BMS 响应时间记录模块（bms_response_log）

| 项目       | 说明                                                                                               |
| -------- | ------------------------------------------------------------------------------------------------ |
| **输入**   | 故障注入成功时的通知（触发类型、通道号、电压值），以及 CAN 接收帧回调                                                            |
| **处理流程** | 收到故障通知 → 记录 T1（触发时间）→ 后续收到的 CAN 帧逐帧检查是否符合过滤条件（ID/mask/数据字节掩码）→ 匹配成功则记录 T2（响应时间）→ 计算 ΔT = T2 - T1 |
| **输出**   | 可通过命令查询的响应记录：T1、T2、延时、响应帧的 ID 和数据                                                                |
| **过滤能力** | 支持标准帧/扩展帧 ID 过滤，支持按数据字节的 bit 掩码过滤                                                                |
| **文件**   | `App/Src/bms_response_log.c`                                                                     |

### 5.8 CAN 通信 BSP 模块（bsp_can）

| 项目       | 说明                                                          |
| -------- | ----------------------------------------------------------- |
| **输入**   | FDCAN1 硬件 RX FIFO 中的 CAN 帧数据                                |
| **处理流程** | 每 1ms 检查 RX FIFO 是否有新帧 → 读取帧 ID、长度、数据 → 更新统计计数 → 触发回调函数通知上层 |
| **输出**   | 向外发送 CAN 帧（通过 TX FIFO），向应用层回调通知接收帧                          |
| **统计信息** | 累计发送/接收/错误计数、最近一帧的 ID/长度/数据/接收时间                            |
| **文件**   | `BSP/Src/bsp_can.c`                                         |

### 5.9 DAC81416 芯片驱动模块

| 项目         | 说明                                                         |
| ---------- | ---------------------------------------------------------- |
| **输入**     | 读写寄存器的请求（寄存器地址、数据值）                                        |
| **处理流程**   | 构造 24 位 SPI 帧 → 通过 SPI1 发送 → 如果是读操作，再发一个 NOP 帧并同时接收返回值     |
| **输出**     | DAC 芯片的寄存器值或 DAC 通道输出电压                                    |
| **SPI 保护** | 所有 SPI 操作受 FreeRTOS 互斥锁保护，多任务不会同时访问 SPI 总线                 |
| **文件**     | `Drivers/DAC81416/Src/dac81416.c`、`BSP/Src/bsp_dac81416.c` |

### 5.10 以太网底层模块（ethernetif）—— 包含 PHY 状态机

| 项目        | 说明                                                                        |
| --------- | ------------------------------------------------------------------------- |
| **输入**    | 以太网 PHY 芯片（LAN8720A）通过 RMII 接口传来的数据包                                      |
| **处理流程**  | 运行 PHY 状态机 → 扫描→复位→检测链路→收发数据。每 100ms 检测一次链路状态，双重读取 BSR 寄存器避免 latch-low 效应 |
| **输出**    | 向 LwIP 协议栈提交接收到的数据包（pbuf），通过 DMA 发送待发送的数据包                                |
| **热插拔支持** | 链路断开时自动清除 PHY 地址缓存，下次插上时重新扫描发现                                            |
| **文件**    | `LWIP/Target/ethernetif.c`                                                |

## 六、数据流：模块之间怎么配合

### 6.1 控制命令的完整路径

```text
你通过串口打字 "ov 7 4500 100 3000"
  │
  ▼
fault_console（串口接收层）
  把收到的字节拼成完整一行
  │
  ▼
fault_command（命令解析层）
  解析文本 → 识别为"过压故障" → 提取参数（通道=7, 目标=4500mV, 持续=100ms, 斜率=3000mV/ms）
  │
  ▼
voltage_sim（电压模拟层）
  记录故障参数到第7路的状态机（设置 type=OV, target=4500, restore=当前电压）
  │
  ▼
defaultTask 的 1ms 循环（每 1ms 执行一次）
  voltage_sim.Process() → 运行13路故障状态机 → 第7路处于上升阶段，逐步升压
  waveform_gen.Process() → 运行波形状态机（如果激活则计算采样点）
  dac_safety.Process()   → 运行安全状态机，检查报警
  bsp_can.Process()      → 检查CAN是否收到新帧
  ↓
  DAC81416 驱动 → SPI 写入 → 芯片输出电压
```

### 6.2 安全保护的完整路径

```text
DAC 芯片检测到异常（nALMOUT 引脚变低电平）
  │
  ▼
dac_safety.Process()（安全状态机检测到引脚变化）
  正常状态 → 报警状态转换
  │
  ▼
dac_safety.EmergencyStop()
  停止波形 → 清除所有故障 → 设置全部 13 路为安全电压（如 3200mV）
  → 设置 emergencyActive = 1（进入急停锁存状态）
  │
  ▼
急停锁存生效
  后续所有 "normal / set / ov / uv / wave" 等命令 → DacSafety_IsOutputAllowed() 返回 0
  → 命令层返回"拒绝，急停中"
  │
  ▼
用户手动执行 "safe release"
  → emergencyActive = 0 → 回到正常状态 → 恢复控制
```

### 6.3 BMS 响应时间记录的完整路径

```text
用户执行 "ov 7 4500"
  │
  ▼
voltage_sim 注入过压故障（第7路状态机进入上升阶段）
  │
  ▼
bms_response_log.RecordFaultTrigger()
  记录 T1 = 当前时间
  │
  ▼
BMS 检测到过压 → 通过 CAN 总线发出告警帧
  │
  ▼
bsp_can 收到 CAN 帧 → BspCan_OnRxFrame() 回调
  │
  ▼
bms_response_log 检查该帧是否符合过滤规则
  ├─ 不符合 → 丢弃，filter_miss 计数 +1
  └─ 符合 → 记录 T2 = 当前时间，保存响应帧的 ID 和数据
       → 计算 ΔT = T2 - T1
       → 用户通过 "log status" 查询结果
```

## 七、FreeRTOS 任务配置（4 个任务）

### 7.1 任务一览表

| 任务名                      | 任务函数                     | 优先级                 | 栈大小     | 运行方式              | 创建时机              |
| ------------------------ | ------------------------ | ------------------- | ------- | ----------------- | ----------------- |
| **defaultTask**          | `StartDefaultTask()`     | 中等（Normal）          | 4096 字节 | 每 1ms 循环一次        | FreeRTOS 初始化时创建   |
| **EthIf**                | `ethernetif_input()`     | **最高（Realtime）**    | 512 字节  | 由信号量触发（收到以太网包时唤醒） | LwIP 初始化时创建       |
| **TcpJson**              | `TcpJsonServer_Task()`   | **最低（BelowNormal）** | 2048 字节 | 阻塞等待 TCP 客户端连接    | defaultTask 启动时创建 |
| **ethernet_link_thread** | `ethernet_link_thread()` | 中等（Normal）          | 256 字节  | 每 100ms 轮询一次      | LwIP 初始化后创建       |

### 7.2 每个任务在做什么

**defaultTask（主任务）—— 最重要，所有状态机都在这里运行**

这个任务是系统的核心，每 1ms 运行一次，依次执行：

1. **电压模拟处理**（~50μs）：运行 13 路故障状态机，计算当前电压值
2. **波形生成处理**（~80μs）：运行波形状态机，计算当前采样点
3. **安全保护检查**（~10μs）：运行安全状态机，读取 DAC 报警引脚
4. **串口输入检查**（~5~500μs）：检查 USART1 是否有新数据
5. **CAN 接收检查**（~10~200μs）：检查 FDCAN1 的 RX FIFO 是否有新帧
6. **心跳日志**（每 1000ms 执行一次）：翻转 LED，打印运行状态

**EthIf（以太网接收任务）—— 优先级最高**

以太网 DMA 收到数据包后触发中断，中断中释放信号量唤醒此任务。任务立即从 DMA 缓冲区取出数据包交给 LwIP 协议栈处理。优先级必须最高，否则网卡缓冲区可能溢出丢包。

**TcpJson（TCP 服务任务）—— 优先级最低**

监听端口 5000，接受客户端连接，读取客户端发来的 JSON 命令，调用命令解析模块执行，结果写回客户端。优先级最低，因为人操作的上位机慢几百毫秒都没关系。

**ethernet_link_thread（网线检测任务）—— 包含 PHY 状态机**

每 100ms 检查一次网线是否插好（运行 PHY 状态机）：

- 检测到网线插入 → 自动配置 MAC 速率和双工模式 → 启动网络收发
- 检测到网线拔出 → 停止网络收发 → 清除 PHY 缓存

### 7.3 同步保护机制

系统中使用 4 个同步对象来防止多任务冲突：

| 同步对象           | 类型  | 保护什么   | 使用场景                        |
| -------------- | --- | ------ | --------------------------- |
| printfMutex    | 互斥锁 | 串口输出   | 多个任务同时调用 `printf` 时防止输出内容交错 |
| dacSpiMutex    | 互斥锁 | SPI 总线 | 多个任务同时访问 DAC 芯片时防止 SPI 数据错乱 |
| RxPktSemaphore | 信号量 | 以太网接收  | 通知 EthIf 任务有新的数据包需要处理       |
| TxPktSemaphore | 信号量 | 以太网发送  | 通知发送任务数据包已发送完成              |

## 八、命令处理流程

当串口或 TCP 收到命令时，处理路径如下：

```text
串口收到 "ov 7 4500 100 3000"（或 TCP 收到 JSON）
  │
  ▼
FaultCommand_ExecuteLine() 统一入口
  │
  ├─ 首字符是 '{' → 按 JSON 解析（如 {"cmd":"ov","cell":7,...}）
  └─ 否则 → 按文本命令解析（如 "ov 7 4500 100 3000"）
  │
  ▼
解析出：命令类型=过压, 通道=7, 目标电压=4500mV, 持续时间=100ms, 斜率=3000mV/ms
  │
  ▼
检查 DacSafety_IsOutputAllowed() → 安全状态机是否在急停中？
  │
  ├─ 正常（未急停）→ 停止当前波形 → 调用 VoltageSim_InjectCellOverVoltageRamp()
  │   → 设置第7路故障状态机的参数（type=OV, target=4500, restore=当前值, 斜率...）
  │   → 故障状态机从"空闲"进入"上升阶段"
  │
  └─ 急停中 → 返回错误信息
  │
  ▼
注入成功 → 自动调用 BmsResponseLog_RecordFaultTrigger() 记录 T1
  │
  ▼
defaultTask 的下一次 1ms 循环，故障状态机开始工作，逐步升压
```

**关键设计原则：** 命令处理是"写入配置，下次循环生效"的模式。收到命令后只是修改故障状态机的参数，实际的电压变化在下一个 1ms 循环中由状态机驱动。这保证了电压输出总是在固定的时间周期内更新。
